# 分支管理规范

## 一、分支命名

**格式**: `类型/范围-名称`

| 类型 | 用途 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat/tui-refactor` |
| `fix` | 修复 | `fix/deepseek-streaming` |
| `refactor` | 重构 | `refactor/boulder-db` |
| `chore` | 辅助（依赖更新等） | `chore/update-deps` |
| `docs` | 文档 | `docs/provider-api` |
| `test` | 测试 | `test/hook-system` |
| `perf` | 性能优化 | `perf/rendering-cache` |

**范围**（从代码库结构中提取）:
`tui`, `providers`, `platform`, `core`, `orchestration`, `integration`, `mcp`

**名称规则**: 小写、连字符分隔、简短描述

---

## 二、工作树命名

**格式**: `.worktrees/{branch-name}` (与分支名完全对应)

```
.worktrees/
├── feat-tui-refactor/      # ← feat/tui-refactor 分支
├── fix-deepseek-streaming/  # ← fix/deepseek-streaming 分支
├── chore-update-deps/       # ← chore/update-deps 分支
└── ...
```

**规则**:
- 工作树目录名 = 分支名（`/` 替换为 `-`）
- 每个分支一个工作树
- 不在主目录下直接开发

---

## 三、分支创建规则

| 类型 | 创建起点 | 说明 |
|------|----------|------|
| `feat` | `main` | 新功能从稳定起点开始 |
| `fix` | 当前工作分支 | 快速修复直接从当前分支创建 |
| `refactor` | `main` 或当前分支 | 根据范围决定 |
| `chore` | `main` | 依赖更新等从稳定分支开始 |
| `docs` | `main` | 文档修改从稳定分支开始 |
| `test` | 当前分支 | 随对应代码创建 |
| `perf` | `main` 或当前分支 | 根据优化范围决定 |

**示例**:
```bash
# feat - 必须从 main
git checkout main && git checkout -b feat/new-feature

# fix - 从当前分支快速创建
git checkout feat/tui-refactor && git checkout -b fix/login-crash

# refactor - 可从 main 或当前分支
git checkout main && git checkout -b refactor/boulder-db
```

---

## 四、提交信息规范

**格式**: `类型(范围): 描述`

```
feat(tui): add debug panel toggle
fix(deepseek): resolve streaming timeout
refactor(platform): simplify hook registration
chore(deps): update ratatui to 0.26
docs(providers): add deepseek streaming spec
test(boulder): add integration test for temp dir
```

**规则**:
- 第一行不超过 72 字符
- 描述用动词开头（add, fix, resolve, simplify 等）
- 第二行空白，第三行起可选详细说明

---

## 五、生命周期规则

| 操作 | 规则 |
|------|------|
| 创建分支 | 从对应起点分支创建 |
| 合并 | 合并后删除源分支 |
| 工作树 | 同步删除已合并分支的工作树 |
| 长期分支 | `main` 是唯一长期分支 |
